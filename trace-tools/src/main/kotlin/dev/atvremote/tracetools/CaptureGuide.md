# Golden-trace capture guide (real Apple TV — deferred)

> **Status: the shipped fixtures are SYNTHETIC.**
> `protocol/src/test/resources/goldentrace/{pair-setup,pair-verify,hid-menu}.json`
> are deterministically generated by the in-repo reference oracle
> `dev.atvremote.tracetools.GoldenTraceGen` (fixed seeds, **no RNG**, no real
> device). They prove protocol-logic self-consistency only. This document tells
> the project owner how to capture **real tvOS 18 traces** later and swap them
> in. The authoritative real-device end-to-end check remains **Task 17** (CLI
> smoke test against a live Apple TV).

---

## 0. Why this is deferred

Task 10 in the plan captures wire traces from a real tvOS 18 device using
[`pyatv`](https://pyatv.dev/). There was no physical Apple TV and no `pyatv`
available when this task ran. The project owner explicitly approved building all
code and stubbing the fixtures with **clearly-flagged synthetic data**, with the
real capture deferred to whoever has hardware.

When you perform the real capture below, the resulting JSON **replaces** the
synthetic files and you set `"mode": "realDevice"` (and update/remove the
synthetic `"note"`). Tasks 11/12 tests (`PairSetup` / `PairVerify` reproduce
`out(i)`) then validate against *real* bytes instead of oracle bytes.

---

## 1. Prerequisites — install pyatv

On a machine on the **same LAN** as the Apple TV:

```bash
python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install pyatv
atvremote --version                # sanity check
```

## 2. Scan for the device

```bash
atvremote scan
```

Note the Apple TV's **Identifier** (a UUID/MAC-like string). Export it:

```bash
export ATV_ID="<paste the identifier from scan>"
```

## 3. Capture pair-setup (PIN pairing)

The Apple TV will display a 4-digit PIN on screen; type it when prompted.

```bash
atvremote --id "$ATV_ID" --protocol companion --debug pair 2> pair.log
```

`pair.log` now contains the full Companion debug log including every frame
sent/received during pair-setup M1–M6.

## 4. Capture pair-verify + an HID command

Pair-verify happens on **every** connection, so any command captures it. Capture
the **Menu** button (HID code 5) to also produce the `hid-menu` trace:

```bash
atvremote --id "$ATV_ID" --protocol companion --debug menu 2> menu.log
```

`menu.log` contains pair-verify M1–M3 followed by the post-handshake E_OPACK HID
Menu frames (button down then up).

---

## 5. Extract frames from the `--debug` logs into JSON

pyatv's Companion debug lines look roughly like:

```
DEBUG: Send Data: FrameType.PS_Start ... <hex bytes>
DEBUG: Received Data: FrameType.PS_Next ... <hex bytes>
```

For each handshake, extract the frames **in protocol order** and produce one
fixture object per file with this exact shape (same schema the synthetic
generator emits, so `GoldenTrace` / Tasks 11/12 keep working unchanged):

```json
{
  "mode": "realDevice",
  "kind": "pair-setup",
  "note": "Captured from <Apple TV model> tvOS <version> via pyatv on <date>.",
  "fixedInputs": {
    "controllerEd25519Seed": "hex:<the 32B auth seed pyatv used for this pairing>",
    "controllerPairingId":   "hex:<controller pairing id bytes>",
    "pin": "<the PIN you typed>",
    "accessoryId": "hex:<atv pairing id bytes>"
  },
  "steps": [
    { "dir": "out", "frameType": 3, "opackHex": "<hex of full OPACK payload>", "decoded": { } },
    { "dir": "in",  "frameType": 4, "opackHex": "<hex>", "decoded": { } },
    { "dir": "out", "frameType": 4, "opackHex": "<hex>", "decoded": { } },
    { "dir": "in",  "frameType": 4, "opackHex": "<hex>", "decoded": { } },
    { "dir": "out", "frameType": 4, "opackHex": "<hex>", "decoded": { } },
    { "dir": "in",  "frameType": 4, "opackHex": "<hex>", "decoded": { } }
  ]
}
```

Mapping rules (must hold or Tasks 11/12 indexing breaks):

| File              | `kind`        | `steps` order (dir/frameType)                                  |
|-------------------|---------------|----------------------------------------------------------------|
| `pair-setup.json` | `pair-setup`  | out·3, in·4, out·4, in·4, out·4, in·4   (M1..M6)                |
| `pair-verify.json`| `pair-verify` | out·5, in·6, out·6                       (M1..M3)               |
| `hid-menu.json`   | `hid-menu`    | out·8, out·8                             (Menu down, Menu up)   |

Notes:

- `opackHex` is the **OPACK payload of the frame body**, i.e. the bytes *after*
  the 4-byte Companion frame header (and, for pair-verify M2/M3 and HID, the
  decrypted plaintext is **not** used — record the on-the-wire OPACK payload
  exactly as the controller would emit / the accessory sent; pair-setup/verify
  payloads are themselves unencrypted OPACK wrapping a TLV8 blob).
- `frameType` integers: `PS_Start=3, PS_Next=4, PV_Start=5, PV_Next=6,
  E_OPACK=8` (see `protocol/.../frame/Frame.kt`).
- `decoded` is optional/advisory for real captures — the loader and Tasks 11/12
  only consume `opackHex`, `dir`, `frameType`, and `fixedInputs`. You may leave
  `decoded` as `{}` for real captures, or populate it for human readability.
- `controllerEd25519Seed` must be the **actual auth private seed pyatv used**
  for that pairing; without it Task 11 cannot reproduce M3/M5 byte-for-byte.
  pyatv stores credentials under `~/.pyatv.conf` (or the path it prints); the
  Companion auth key is the seed. If you cannot recover the exact seed, the
  real fixture can still validate **structure** (Task 17 is the true E2E gate)
  but not byte-exact controller reproduction — flag that in `"note"`.
- For `pair-verify.json` also include `controllerX25519Priv` /
  `controllerX25519Pub` (the ephemeral pair-verify key, if recoverable from the
  debug log) plus the persisted creds (`clientLtpk`, `atvLtpk`,
  `controllerPairingId`, `accessoryId`) so `GoldenTrace.credentials` /
  `fixedX25519*` resolve.

## 6. Swap in & validate

1. Overwrite the three files in
   `protocol/src/test/resources/goldentrace/` with your real captures
   (`"mode": "realDevice"`).
2. `./gradlew :protocol:test --console=plain` — `GoldenTraceLoaderTest` will
   need its `assertEquals("synthetic", gt.mode)` expectations relaxed to accept
   `"realDevice"`; update that test accordingly when you flip the mode.
3. Run Tasks 11/12 tests — they must reproduce `out(i)` against the real bytes.
4. Run **Task 17** CLI smoke test against the live device for the authoritative
   end-to-end confirmation.

## 7. Regenerating the SYNTHETIC fixtures (until real capture exists)

```bash
./gradlew :trace-tools:runGoldenTraceGen
# convenience alias:
./gradlew :trace-tools:run -PgenGolden
```

This runs the oracle's full synthetic pair-setup + pair-verify + HID handshake
end-to-end (SRP proofs, Ed25519 signatures, ChaCha20-Poly1305 decrypt, HKDF
connection keys) and **aborts before writing** if anything is internally
inconsistent. Output is byte-identical on every run (fixed seeds, no RNG); a
second run followed by `git diff` should show no change.
